01how it works

A deny-list, not an LLM judge

Every check is deterministic — pattern walls plus an independent delete-analyzer that resolves what a command would actually destroy. No model, no temperature, nothing to persuade: same command, same verdict, every time. Engine error or unparseable input? It blocks. Unchecked is never treated as safe.

step 1

The hook fires first

Your agent proposes a command. Before a single byte executes, the harness hands it to gate.cat (a PreToolUse hook). The model can't skip a step it doesn't control — the veto lives outside its reach.

step 2

Throwaway env? Disarm

In a throwaway CI sandbox nothing is irreversible, so the gate disarms and logs a no-op. On a real machine it stays armed.

step 3

The delete-analyzer

Not a regex: for delete-shaped commands it resolves the actual path that would be destroyed and checks it against protected roots — so rm -rf ~/projects/app is caught by what it deletes, not by how it's spelled.

step 4

71 default policy walls → verdict

Deterministic patterns for the irreversible class, first match wins.

ALLOW — runs untouched (~99.4%)
WARN — ambiguous, a human decides
BLOCK — exit 2, the command never runs

02use it

Works where your agent runs

The veto has to sit at the point of execution. Pick the one that matches your setup. The first two need zero code.

Claude Code hook

Enforcement in the harness

A PreToolUse hook fires before every Bash call, outside the model's control flow. The model cannot talk its way past it.

// .claude/settings.json "PreToolUse": [{ "matcher": "Bash", "hooks": [{ "type": "command", "command": "gatecat-hook" }]}]
Proxy · any OpenAI-API agent

One base_url, zero code

GitHub Copilot (BYOK), Ollama, vLLM — anything that speaks the OpenAI API but doesn't run its own shell. Point its base_url at the proxy and it vetoes dangerous tool-calls before your agent runs them.

export OPENAI_BASE_URL=…:11434/v1 gatecat-proxy # :8080 # point agent → :8080/v1
Gated shell · Codex, aider & CLI agents

Hook-strength for any CLI agent

Codex, aider, OpenClaw, Hermes, Antigravity — anything that runs shell commands. Route the tool's shell through gatecat-shell and every command is gated at the exec point, outside the model — the same can't-talk-its-way-past enforcement as the Claude Code hook.

gatecat-shell -c "wipefs -af /dev/sda" # → veto, exit 2 — never reaches the disk
Python SDK

One line in your loop

Own the agent code? Guard the action before you execute it. Works with any model, local or frontier.

from gatecat.integrations \ import check_action, DOGFOOD_DEFAULTS check_action("agent", cmd, DOGFOOD_DEFAULTS) # raises if bad

The honest line, up front

The gate is certain only about what it blocks. An action it does not match is unchecked, not safe. This is defense-in-depth and a fail-safe, not a promise to catch every bad action. It's a deterministic deny-list, so it catches known dangerous shapes; a novel destructive command it has never seen can slip past.

The bypass suite ships in the repo and publishes its own remaining gap — a command whose binary name is assembled at runtime, which no pattern layer can see. Software is provided AS IS; you remain responsible for your agent's actions. Apache 2.0.

07faq

Questions people actually ask

Is it really free?

Yes. The core — the gate, all 71 default policies, the Claude Code hook, the proxy, the gated shell, the CLI, local reports — is free forever, Apache 2.0, no trial, no rate limit. Paid things are convenience and proof (stack-specific policy packs, the off-machine Cloud copy). Safety is never behind a paywall — when we find a universal catastrophic gap, the fix goes into the free core.

Does it work with my agent?

Claude Code gets a native hook — enforcement in the harness, outside the model's control. CLI agents that run a shell — Codex, aider, OpenClaw, Hermes, Antigravity — get the same out-of-model enforcement via the gated shell: route the tool's shell through gatecat-shell and every command is gated at the exec point, it can't talk its way past. Anything that only speaks the OpenAI API — GitHub Copilot (BYOK), Ollama, vLLM — can instead go through the proxy: point its base_url at gate.cat to veto tool-calls. Own the agent code? One line with the Python SDK.

Can't the agent just talk its way past it?

Not past the hook. It's a deterministic deny-list that runs before the tool call, outside the model's control flow — same command, same verdict, every time. There's no temperature and nothing to persuade. It also fails closed: engine error or weird input → block. Honest caveat: the in-process framework adapters (crewAI/LangGraph) are a weaker trust class — a prompt injection can route around them. Only the hook is real enforcement, and we label which is which.

What does it NOT catch?

The gate is certain only about what it blocks — an action it doesn't match is unchecked, not safe. It's a deterministic deny-list of known dangerous shapes, so what it can't catch is the obfuscated or novel command — a binary name assembled at runtime, a payload built from pieces — and that's exactly what the bypass suite in the repo publishes instead of hiding. It's also not a fact-checker — don't buy it for hallucination detection. A measured 38-policy historical snapshot and the current limitations are on the coverage page, along with a no-account form to report a command we miss.

Will it nag me all day?

No — that's the design constraint. Measured on real traffic (a 14.7k Claude Code log and a public 8.6k SWE-agent corpus), it intervenes on ~0.6% of commands. A guardrail that nags gets turned off, so the walls are tuned to fire on the irreversible class, not on your everyday work.

What data leaves my machine?

Nothing. The pip package phones nowhere — zero telemetry, no analytics. Cloud is opt-in and off by default; even then the default payload is a hash of the matched command, not the text (raw text is a separate explicit opt-in, because commands can contain secrets).
